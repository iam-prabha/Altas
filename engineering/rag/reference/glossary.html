RAG Glossary

Key terms introduced across lessons, defined for quick reference.

A

Augmentation
The step where retrieved document chunks are combined with the user's query to form a prompt for the LLM. The "Augmented" in RAG.

C

Chunk
A small piece of text that a document is split into during indexing. Chunks should be large enough to contain meaningful information but small enough to fit in the LLM's context window.
Chunk Overlap
The number of characters shared between consecutive chunks to prevent information loss at chunk boundaries. Typical overlap: 10–20% of chunk size.
Chunk Size
The maximum length (in characters or tokens) of each chunk. Common sizes: 256–1024 tokens depending on the use case.
Context Window
The maximum number of tokens an LLM can process in a single request. GPT-4o-mini has a 128k token context window; smaller models may have 4k–32k. Larger context windows reduce the need for aggressive chunking but increase cost and latency.
Cosine Similarity
A measure of similarity between two vectors (range: -1 to 1) calculated as the cosine of the angle between them. Commonly used in text embedding search instead of L2 distance.

E

Embedding
A dense vector (list of floating-point numbers) that represents the semantic meaning of a piece of text. Similar texts produce vectors close to each other in the vector space. Created by an embedding model (e.g., all-MiniLM-L6-v2).
Embedding Model
A neural network trained to map text to vectors such that semantically similar texts are close in vector space. Popular models: all-MiniLM-L6-v2 (384 dimensions), BAAI/bge-large-en-v1.5 (1024 dimensions), OpenAI text-embedding-3-small.

F

FAISS
Facebook AI Similarity Search — a library for efficient vector similarity search and clustering. Supports L2, cosine, and dot-product distance metrics, with GPU acceleration and disk-based indexes for billion-scale datasets.

H

Hallucination
When an LLM generates factually incorrect or fabricated information. RAG reduces hallucinations by grounding the model's output in retrieved evidence.
Hybrid Search
A retrieval strategy that combines vector (semantic) search with keyword (lexical) search to get the best of both approaches. Typical approach: BM25 + vector search with reciprocal rank fusion.

I

Index
(Verb) The process of preparing documents for retrieval: chunking + embedding + storing vectors. (Noun) The data structure (e.g., FAISS index) that enables fast similarity search.

L

L2 Distance
Euclidean distance between two vectors. The most basic similarity metric. Smaller L2 distance = more similar vectors.
LLM
Large Language Model — the text generation component in RAG. Given a prompt with context, it produces a grounded answer.

N

Naive RAG
The simplest RAG architecture: Index → Retrieve → Generate in a straight line. No query rewriting, reranking, or iterative retrieval. The starting point for all RAG systems.

O

Ollama
A tool for running LLMs locally. Supports models like Llama 3.2, Mistral, and Qwen. Exposes a REST API at localhost:11434.

R

RAG
Retrieval-Augmented Generation — a system architecture where an LLM generates answers grounded in documents retrieved from an external knowledge base. Reduces hallucinations and enables access to up-to-date information.
Reranking
A post-retrieval step where initially retrieved chunks are re-scored by a more accurate (but slower) model to improve result quality. Common in Advanced RAG pipelines.
Retrieval
The step of finding the most relevant document chunks for a given query. Typically done by embedding the query and performing nearest-neighbor search in the vector index.

S

Sentence Transformers
A Python library providing pre-trained embedding models. SentenceTransformer class loads models and encodes text into vectors.
Similarity Search
The core retrieval operation: given a query vector, find the closest vectors in the index by some distance metric (L2, cosine, dot product).

V

Vector
A list of numbers (e.g., 384 floats) representing a piece of text in semantic space. Generated by an embedding model.
Vector Database
A database optimized for storing and searching vectors. Examples: FAISS (in-memory), Chroma, Pinecone, Weaviate, Qdrant, pgvector. Chosen based on scale, latency, and operational requirements.
