Lesson 3: Generics

Lesson 3 of ? · ~25 minutes

So far every function and type has worked with one specific type or a union of known types. Generics let you write code that works with any type while keeping the type system fully informed.

Mental model: A generic is a type parameter — a placeholder for a real type that gets filled in at the call site. Think of it like a function parameter, but for types instead of values.

Generic Functions

The syntax: angle brackets <T> before the parameter list introduce a type variable.

function identity<T>(value: T): T {
  return value;
}

const a = identity("hello");  // T inferred as string
const b = identity(42);       // T inferred as number

TypeScript infers T from the argument you pass. You can also specify it explicitly:

const c = identity<string>("hello");  // explicit — same result

Generic arrow functions use the same syntax:

const wrap = <T>(value: T): { value: T } => ({ value });

const wrapped = wrap(99);  // { value: number }

Multiple Type Parameters

You can have more than one. Convention uses T, U, V.

function pair<T, U>(first: T, second: U): [T, U] {
  return [first, second];
}

const p = pair("age", 30);  // type: [string, number]

Generic Types & Interfaces

Type aliases and interfaces can also carry type parameters:

type Box<T> = { value: T };

interface Repository<T> {
  get(id: string): T | undefined;
  set(id: string, value: T): void;
}

Usage:

const numBox: Box<number> = { value: 42 };
const strBox: Box<string> = { value: "hello" };

Constrained Generics with extends

Sometimes you need to restrict what types are allowed. Use extends to constrain the type parameter to a subset. 

function getLength<T extends { length: number }>(value: T): number {
  return value.length;
}

getLength("hello");   // 5 — string has .length
getLength([1, 2, 3]); // 3 — array has .length
// getLength(42);     // Error! number has no .length

You can also constrain to a specific shape:

interface HasId {
  id: string;
}

function findById<T extends HasId>(items: T[], id: string): T | undefined {
  return items.find(item => item.id === id);
}
Naming: T is conventional for single-parameter generics, but use descriptive names (TItem, TResult) when clarity matters. Prefer single uppercase letters only for trivial cases.

Practical Pattern: Result<T, E>

A common production pattern is a Result type that represents either success or failure — no exceptions, no null ambiguity. 

type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function succeed<T>(value: T): Result<T> {
  return { ok: true, value };
}

function fail<E = Error>(error: E): Result<never, E> {
  return { ok: false, error };
}

Because Result is a discriminated union on ok, TypeScript narrows it automatically in if blocks: 

const result = Math.random() > 0.5
  ? succeed("it worked")
  : fail(new Error("it failed"));

if (result.ok) {
  console.log(result.value);  // narrowed — safe to access
} else {
  console.error(result.error.message);
}
